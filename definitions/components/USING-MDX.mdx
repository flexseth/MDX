# Using MDX Component Examples

These MDX files are **copy-paste blueprints** for building WordPress blocks. Each file contains complete, working code that you can drop directly into a `@wordpress/create-block` project. This guide walks you through the process from scaffold to working block, starting with the simplest component and building up to a real-world multi-component block.

## Prerequisites

Before starting, make sure you have:

- **Node.js 18+** installed (`node -v` to check)
- A **local WordPress environment** (wp-env, Local, MAMP, etc.)
- **Terminal access** to run npm commands
- A WordPress site with the **block editor enabled**

## Understanding the File Structure

Each MDX file maps directly to files in a `@wordpress/create-block` project. Here is how the sections translate:

| MDX Section | Destination File |
|---|---|
| "block.json Attributes" | `src/block-name/block.json` |
| "edit.js" examples | `src/block-name/edit.js` |
| "save.js" examples | `src/block-name/save.js` |
| "Required Imports" | Top of `edit.js` or `save.js` |
| "Props Reference" | Development reference (not copied) |

The scaffolded project structure looks like this:

```
my-block/
├── build/                  ← Generated by npm run build
├── src/
│   └── my-block/
│       ├── block.json      ← Attributes, metadata
│       ├── edit.js         ← Editor UI (copy examples here)
│       ├── save.js         ← Front-end output (copy save examples here)
│       ├── index.js        ← Block registration (usually untouched)
│       ├── editor.scss     ← Editor-only styles
│       └── style.scss      ← Shared styles (editor + front-end)
├── my-block.php            ← Plugin bootstrap
├── package.json
└── readme.txt
```

## Step 1: Scaffold a Block

Run the scaffolding command from your WordPress plugins directory:

```bash
cd wp-content/plugins
npx @wordpress/create-block@latest my-button-block
```

This creates a fully working block plugin. Activate it immediately:

```bash
cd my-button-block
npm run build
```

Then activate "My Button Block" in **Plugins > Installed Plugins** in your WordPress admin.

## Step 2: Choose a Component

Start with **Button** (`Button.mdx`). It is the simplest component in this collection because:

- It requires **no block attributes** in `block.json`
- It is **editor-only** (no custom `save.js` needed)
- It has a single import from `@wordpress/components`
- The code is short and easy to verify

Once you understand the copy-paste workflow with Button, you can apply the same process to any component.

## Step 3: Replace edit.js

Open `Button.mdx` and find the "Basic Usage in edit.js" section. Copy that code into your scaffolded `edit.js`, replacing `'your-text-domain'` with `'my-button-block'`.

Your `src/my-button-block/edit.js` should now contain:

```javascript
import { __ } from '@wordpress/i18n';
import { Button } from '@wordpress/components';
import { useBlockProps } from '@wordpress/block-editor';

export default function Edit() {
	const blockProps = useBlockProps();

	const handleClick = () => {
		alert( __( 'Button clicked!', 'my-button-block' ) );
	};

	return (
		<div { ...blockProps }>
			<Button
				variant="primary"
				onClick={ handleClick }
			>
				{ __( 'Click Me', 'my-button-block' ) }
			</Button>
		</div>
	);
}
```

That is a complete, working edit.js. No other changes are needed.

## Step 4: Update save.js (When Needed)

Button is an editor-only interaction, so the default `save.js` works as-is. But many components **do** need a custom `save.js` to render their content on the front-end.

Use this table to decide:

| Component | Needs custom save.js? | Reason |
|---|---|---|
| InspectorControls | No | Sidebar only; stored values applied in save.js |
| BlockControls | No | Toolbar only; stored values applied in save.js |
| Button | No | Editor interaction only |
| TextControl | Yes | Displays saved text on front-end |
| TextareaControl | Yes | Displays saved multi-line text on front-end |
| CheckboxControl | Depends | If booleans control front-end visibility or class names |
| RadioControl | Depends | If selection controls front-end layout or class names |
| RichText | Yes | Displays formatted content with `RichText.Content` |
| MediaUpload | Yes | Displays uploaded image/media |
| FocalPointPicker | Yes | Applies `background-position` or `object-position` |
| RangeControl | Depends | If the value controls styling, save.js applies it |
| SelectControl | Depends | If the selection changes front-end output |
| ComboboxControl | Depends | If the selected value changes front-end content |
| FormTokenField | Yes | Token list displayed as tags/list on front-end |
| ColorPalette | Depends | If the color applies to front-end elements |
| ColorPicker | Depends | If the color applies to front-end elements |
| FontSizePicker | Yes | Font size applied to front-end text |
| UnitControl | Yes | CSS dimension applied to front-end styles |
| BoxControl | Yes | Padding/margin applied to front-end styles |
| ToggleControl | Depends | If the toggle controls front-end visibility |
| PanelBody | No | Sidebar organization only |
| TabPanel | No | Editor-only tabbed interface |
| Dropdown | No | Editor-only popover |
| Card | No (usually) | Editor UI; use semantic HTML in save.js |
| Notice | No | Editor-only status message |
| Spinner | No | Editor-only loading indicator |
| Modal | No | Editor-only dialog |
| Tooltip | No | Editor-only hover text |
| Placeholder | No | Editor-only empty state (return null when unconfigured) |
| Guide | No | Editor-only onboarding modal |
| DateTimePicker | Depends | If the date/time displays on front-end |

**Rule of thumb:** If a component stores data that a visitor should see, you need a custom `save.js`. If it only controls the editor experience, the default save is fine.

## Step 5: Add Attributes to block.json

Button has no attributes, so `block.json` stays as scaffolded. But most components need attributes to store their data. Here is how to add them, using TextControl as the example.

### Before (scaffolded default)

Your `src/my-button-block/block.json` has no `attributes` key:

```json
{
	"$schema": "https://schemas.wp.org/trunk/block.json",
	"apiVersion": 3,
	"name": "create-block/my-button-block",
	"version": "0.1.0",
	"title": "My Button Block",
	"category": "widgets",
	"icon": "smiley",
	"description": "Example block scaffolded with Create Block tool.",
	"example": {},
	"supports": {
		"html": false
	},
	"textdomain": "my-button-block",
	"editorScript": "file:./index.js",
	"editorStyle": "file:./index.css",
	"style": "file:./style-index.css"
}
```

### After (with TextControl attributes)

Add the `attributes` key from `TextControl.mdx`:

```json
{
	"$schema": "https://schemas.wp.org/trunk/block.json",
	"apiVersion": 3,
	"name": "create-block/my-button-block",
	"version": "0.1.0",
	"title": "My Button Block",
	"category": "widgets",
	"icon": "smiley",
	"description": "Example block scaffolded with Create Block tool.",
	"example": {},
	"supports": {
		"html": false
	},
	"textdomain": "my-button-block",
	"editorScript": "file:./index.js",
	"editorStyle": "file:./index.css",
	"style": "file:./style-index.css",
	"attributes": {
		"textValue": {
			"type": "string",
			"default": ""
		}
	}
}
```

The `attributes` object from any MDX file merges into your existing `block.json` exactly like this. Each MDX file shows only the attributes it needs; you combine them when using multiple components.

## Step 6: Build and Test

After making changes, rebuild the block:

```bash
npm run build
```

Or start the development watcher for live rebuilds:

```bash
npm start
```

Then verify in WordPress:

1. Go to any post or page in the block editor
2. Click the **+** inserter and search for your block name
3. Insert the block and confirm it renders correctly
4. Check the front-end by previewing the post

If you see a block validation error after making changes, delete the existing block instance in the editor and insert a fresh one. Validation errors occur when the saved HTML no longer matches what `save.js` produces.

## Combining Multiple Components

Real blocks use several components together. Here is a **Team Member Card** block that pulls patterns from five MDX files:

- **RichText** (`RichText.mdx`) for the name and bio
- **MediaUpload** (`MediaUpload.mdx`) for the avatar
- **TextControl** (`TextControl.mdx`) for the job title (sidebar)
- **PanelBody** (`PanelBody.mdx`) for sidebar organization
- **ToggleControl** (`ToggleControl.mdx`) for show/hide social links

### block.json attributes

```json
{
	"attributes": {
		"name": {
			"type": "string",
			"source": "html",
			"selector": "h3"
		},
		"bio": {
			"type": "string",
			"source": "html",
			"selector": "p.team-member-bio"
		},
		"jobTitle": {
			"type": "string",
			"default": ""
		},
		"avatarId": {
			"type": "number",
			"default": 0
		},
		"avatarUrl": {
			"type": "string",
			"default": ""
		},
		"avatarAlt": {
			"type": "string",
			"default": ""
		},
		"showSocialLinks": {
			"type": "boolean",
			"default": false
		}
	}
}
```

### edit.js

```javascript
import { __ } from '@wordpress/i18n';
import { RichText, MediaUpload, MediaUploadCheck, useBlockProps, InspectorControls } from '@wordpress/block-editor';
import { PanelBody, TextControl, ToggleControl, Button } from '@wordpress/components';

export default function Edit( { attributes, setAttributes } ) {
	const blockProps = useBlockProps();
	const { name, bio, jobTitle, avatarId, avatarUrl, avatarAlt, showSocialLinks } = attributes;

	const onSelectAvatar = ( media ) => {
		setAttributes( {
			avatarId: media.id,
			avatarUrl: media.url,
			avatarAlt: media.alt,
		} );
	};

	const onRemoveAvatar = () => {
		setAttributes( {
			avatarId: 0,
			avatarUrl: '',
			avatarAlt: '',
		} );
	};

	return (
		<>
			<InspectorControls>
				<PanelBody
					title={ __( 'Team Member Settings', 'team-member-card' ) }
					initialOpen={ true }
				>
					<TextControl
						label={ __( 'Job Title', 'team-member-card' ) }
						value={ jobTitle }
						onChange={ ( value ) => setAttributes( { jobTitle: value } ) }
						help={ __( 'Displayed below the team member name.', 'team-member-card' ) }
					/>
					<ToggleControl
						label={ __( 'Show Social Links', 'team-member-card' ) }
						checked={ showSocialLinks }
						onChange={ ( value ) => setAttributes( { showSocialLinks: value } ) }
						help={ __( 'Display social media links below the bio.', 'team-member-card' ) }
					/>
				</PanelBody>
			</InspectorControls>

			<div { ...blockProps }>
				<div className="team-member-card">
					<MediaUploadCheck>
						<MediaUpload
							onSelect={ onSelectAvatar }
							allowedTypes={ [ 'image' ] }
							value={ avatarId }
							render={ ( { open } ) => (
								<div className="team-member-avatar">
									{ ! avatarUrl ? (
										<Button variant="secondary" onClick={ open }>
											{ __( 'Upload Avatar', 'team-member-card' ) }
										</Button>
									) : (
										<>
											<img src={ avatarUrl } alt={ avatarAlt } />
											<Button variant="link" onClick={ open }>
												{ __( 'Replace', 'team-member-card' ) }
											</Button>
											<Button variant="link" isDestructive onClick={ onRemoveAvatar }>
												{ __( 'Remove', 'team-member-card' ) }
											</Button>
										</>
									) }
								</div>
							) }
						/>
					</MediaUploadCheck>

					<RichText
						tagName="h3"
						value={ name }
						onChange={ ( value ) => setAttributes( { name: value } ) }
						placeholder={ __( 'Team member name...', 'team-member-card' ) }
						allowedFormats={ [ 'core/bold', 'core/italic' ] }
					/>

					{ jobTitle && (
						<p className="team-member-title">{ jobTitle }</p>
					) }

					<RichText
						tagName="p"
						className="team-member-bio"
						value={ bio }
						onChange={ ( value ) => setAttributes( { bio: value } ) }
						placeholder={ __( 'Write a short bio...', 'team-member-card' ) }
					/>

					{ showSocialLinks && (
						<div className="team-member-social">
							<p>{ __( 'Social links will appear here.', 'team-member-card' ) }</p>
						</div>
					) }
				</div>
			</div>
		</>
	);
}
```

### save.js

```javascript
import { RichText, useBlockProps } from '@wordpress/block-editor';

export default function save( { attributes } ) {
	const blockProps = useBlockProps.save();
	const { name, bio, jobTitle, avatarUrl, avatarAlt, showSocialLinks } = attributes;

	return (
		<div { ...blockProps }>
			<div className="team-member-card">
				{ avatarUrl && (
					<div className="team-member-avatar">
						<img src={ avatarUrl } alt={ avatarAlt } />
					</div>
				) }

				<RichText.Content tagName="h3" value={ name } />

				{ jobTitle && (
					<p className="team-member-title">{ jobTitle }</p>
				) }

				<RichText.Content
					tagName="p"
					className="team-member-bio"
					value={ bio }
				/>

				{ showSocialLinks && (
					<div className="team-member-social">
						{ /* Social links can be extended here */ }
					</div>
				) }
			</div>
		</div>
	);
}
```

Notice the patterns: `RichText` uses `RichText.Content` in save.js, `MediaUpload` renders a plain `<img>` in save.js, `TextControl` values render as plain HTML, and `ToggleControl` / `PanelBody` have no save.js output of their own.

## Which Component to Use?

### ColorPalette vs ColorPicker

| Situation | Use |
|-----------|-----|
| Choosing from the active theme's brand colors | `ColorPalette` |
| Choosing any arbitrary hex/RGB/RGBA color | `ColorPicker` |
| Full WordPress color workflow (palette + custom) | `ColorGradientSettingsDropdown` from `@wordpress/block-editor` |

`ColorPalette` reads from `theme.json` and renders swatches. `ColorPicker` renders a spectrum canvas and numeric inputs. For most blocks, `ColorPalette` is the right choice because it respects theme branding. Use `ColorPicker` only when the user genuinely needs an unrestricted color choice.

---

### CheckboxControl vs RadioControl vs ToggleControl

| Situation | Use |
|-----------|-----|
| Single on/off setting (most common) | `ToggleControl` |
| Multiple independent boolean settings | `CheckboxControl` (one per option) |
| Multiple values from a list where any combination is valid | `CheckboxControl` with array attribute |
| Exactly one value from a set of mutually exclusive options | `RadioControl` |

**Decision tree:** Can more than one option be selected at a time? Yes → `CheckboxControl`. Only one? How many options? 1 → `ToggleControl`. 2–6 options → `RadioControl`. 7+ options → `SelectControl`.

---

### Modal vs Guide

| Situation | Use |
|-----------|-----|
| Single dialog (confirmation, settings form, picker) | `Modal` |
| Multi-step onboarding walkthrough | `Guide` |
| The content fits in one screen | `Modal` |
| The experience spans multiple pages/steps | `Guide` |

`Guide` renders a series of pages the user steps through with "Previous"/"Next" buttons. `Modal` renders a single dialog. For block editor onboarding, `Guide` is the canonical WordPress pattern.

---

### Placeholder vs Spinner

| Situation | Use |
|-----------|-----|
| Block has no configuration yet (initial empty state) | `Placeholder` |
| Block is loading data asynchronously | `Spinner` |
| Block is configured but data is fetching | `Spinner` |
| Block needs the user to take an action to proceed | `Placeholder` |

Use `Placeholder` when the user needs to do something (upload, connect, configure). Use `Spinner` when the block is waiting for something to finish on its own.

---

### TextControl vs TextareaControl

| Situation | Use |
|-----------|-----|
| Single line of text (title, URL, API key, short label) | `TextControl` |
| Multiple lines of text (description, CSS, JSON, excerpt) | `TextareaControl` |
| Formatted / rich text that saves to front-end HTML | `RichText` |
| User input you want to show in a popover with formatting | `RichText` |

**Rule of thumb:** If the value will ever wrap to a second line naturally, use `TextareaControl`. If it is always a single value on one line, use `TextControl`.

---

### ComboboxControl vs SelectControl

| Situation | Use |
|-----------|-----|
| Short list (under ~10 options) | `SelectControl` |
| Long list where users need to search/filter | `ComboboxControl` |
| Options loaded asynchronously (posts, taxonomy terms) | `ComboboxControl` |
| Options are fixed and short | `SelectControl` |

`SelectControl` renders as a native `<select>`. `ComboboxControl` renders as a text input + filtered dropdown. Use `SelectControl` for small static lists — it has lower cognitive overhead. Switch to `ComboboxControl` when scrolling through options becomes impractical.

## Common Patterns

These patterns appear across nearly every MDX file. Recognizing them makes it easier to combine components.

### useBlockProps Pattern

Every `edit.js` uses `useBlockProps()` and every `save.js` uses `useBlockProps.save()`. This is required by the block editor for proper block wrapper rendering.

```javascript
// edit.js
const blockProps = useBlockProps();
return <div { ...blockProps }>{ /* content */ }</div>;

// save.js
const blockProps = useBlockProps.save();
return <div { ...blockProps }>{ /* content */ }</div>;
```

### setAttributes Pattern

Components that store data destructure `attributes` and `setAttributes` from props. Every `onChange` callback follows the same shape:

```javascript
export default function Edit( { attributes, setAttributes } ) {
	const { myValue } = attributes;

	return (
		<SomeControl
			value={ myValue }
			onChange={ ( value ) => setAttributes( { myValue: value } ) }
		/>
	);
}
```

### InspectorControls + PanelBody Pattern

Sidebar settings always follow this structure. Wrap `PanelBody` inside `InspectorControls`, and place your controls inside the panel:

```javascript
import { InspectorControls } from '@wordpress/block-editor';
import { PanelBody } from '@wordpress/components';

return (
	<>
		<InspectorControls>
			<PanelBody title={ __( 'Settings', 'my-text-domain' ) }>
				{ /* Controls go here */ }
			</PanelBody>
		</InspectorControls>
		<div { ...blockProps }>
			{ /* Block content */ }
		</div>
	</>
);
```

### i18n Pattern

All user-facing strings use `__()` from `@wordpress/i18n`. Replace `'your-text-domain'` in every MDX example with your actual block's text domain (found in `block.json` under `"textdomain"`):

```javascript
import { __ } from '@wordpress/i18n';

// Every label, placeholder, help text, and button string:
__( 'My Label', 'my-actual-text-domain' )
```

### Attribute Source Pattern

Attributes with `"source": "html"` extract their value from the saved HTML markup. Attributes without a `source` are stored in the block's comment delimiter. Understanding the difference is key:

```json
{
	"attributes": {
		"content": {
			"type": "string",
			"source": "html",
			"selector": "p"
		},
		"fontSize": {
			"type": "number",
			"default": 16
		}
	}
}
```

- `content` is extracted from a `<p>` tag in the saved HTML (used by RichText)
- `fontSize` is stored in the block comment `<!-- wp:my-block {"fontSize":16} -->` (used by RangeControl, TextControl, etc.)

**Use `"source": "html"` with `"selector"`** for RichText components. **Omit `source`** for form controls like TextControl, RangeControl, ToggleControl, and SelectControl.

## Troubleshooting

### Block Validation Error

**Symptom:** "This block contains unexpected or invalid content" message in the editor.

**Cause:** The saved HTML in the database does not match what `save.js` currently produces.

**Fix:** Delete the block instance and insert a fresh one. This is expected after changing `save.js`.

### Wrong Import Package

**Symptom:** `undefined is not a function` or component does not render.

**Cause:** Importing from the wrong package. The two packages are:

- `@wordpress/components` — Button, TextControl, RangeControl, ToggleControl, SelectControl, ColorPalette, PanelBody, Guide, DateTimePicker
- `@wordpress/block-editor` — RichText, MediaUpload, MediaUploadCheck, InspectorControls, useBlockProps

**Fix:** Check the "Required Imports" section at the top of each MDX file.

### Missing Attributes

**Symptom:** Component value is always `undefined`, changes are not saved.

**Cause:** The attribute is not defined in `block.json`.

**Fix:** Copy the "block.json Attributes" section from the MDX file and merge it into your `block.json`.

### Text Domain Mismatch

**Symptom:** Translations do not load, or linting tools warn about text domain.

**Cause:** The `__()` calls use a different text domain than what is in `block.json`.

**Fix:** Every MDX file uses `'your-text-domain'` as a placeholder. Replace all instances with the `"textdomain"` value from your `block.json`.

### Changes Not Appearing

**Symptom:** You edited a file but the block looks the same.

**Cause:** The build is stale.

**Fix:** Run `npm run build` after every change, or use `npm start` for automatic rebuilds during development.

### MediaUpload Not Showing

**Symptom:** The upload button does not appear.

**Cause:** Missing `MediaUploadCheck` wrapper, which checks user permissions.

**Fix:** Always wrap `MediaUpload` inside `MediaUploadCheck` as shown in `MediaUpload.mdx`.
